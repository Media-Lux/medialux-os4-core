# 1. Optimierungsfelder

## 1.1 Architektur
- Minimalprinzip: nur essentielle Dateien behalten  
- Alle Systembausteine gehören in definierte OS4-Ordner  
- Keine Logik außerhalb von Master-Files  
- Keine Steuerbefehle im Chat speichern (Boundary-Regel)  

## 1.2 KI-Verhalten & Routing
- alle Avatare müssen Behavior V2 verwenden  
- PRIO-Engine ist Pflicht (PRIO 0 – kritisch, PRIO 1 – essenziell ...)  
- Brain2Chain MUSS die Konsistenz aller Systembausteine prüfen  
- VersionGuard MUSS vor jeder Ausführung die Versionslage prüfen  

## 1.3 Dateiorganisation
- Dateien MÜSSEN gemäß der Storage Map gespeichert werden  
- Jede neue Datei MUSS commit-bereit und versioniert sein  
- Keine „losen“ Dateien außerhalb von `/OS4_*` Ordnern

# 2. Harte Systemregeln (non-negotiable)

### Regel 1 – Keine Redundanz
Derselbe Inhalt darf nicht in mehreren Dateien liegen.  
→ Konsolidierung in Master-Files.

### Regel 2 – Keine Chat-Speicherung
Systemwissen gehört ausschließlich in GitHub.  
→ Chat dient nur als Interface, niemals als Quelle.

### Regel 3 – Keine veralteten Dateien im Systemfluss
Veraltete Dateien müssen:
- modernisiert  
oder  
- archiviert  
werden.

### Regel 4 – Migration = Pflicht
Neue Systemmodule oder Regeln müssen durch:
- OS4_Migration_Guide_V2  
- OS4_BackupBatch_Rule_V2  
laufen.

### Regel 5 – Verhalten > Inhalt
Bevor Inhalte erzeugt werden, muss das Verhalten klar sein.

# 3. Performance-Optimierungen

### 3.1 Prioritäten
- PRIO 0: Startlogik, Behavior, Brain2Chain, VersionGuard  
- PRIO 1: Ecosystem, Packs, Events  
- PRIO 2: Erweiterungen  
- PRIO 3: Archiv  

### 3.2 Routing 
Avatare lösen Aufgaben nach:
- Zuständigkeit  
- PRIO  
- Behavior-Regeln  

### 3.3 Auto-Optimierung
Brain2Chain schlägt automatisch vor:
- zu viele Dateien → zusammenführen  
- fehlende Dateien → erstellen  
- veraltete Dateien → modernisieren

# 4. Integration in Brain2Chain & VersionGuard

### VersionGuard MUSS:
